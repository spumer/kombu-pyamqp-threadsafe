## 2. Тезисы (scope A.2.6 + анти-тезис NQD ≥3 B.5.2.1 + тип связи E.4.PFR)

### Тезис 1 — Race condition ⊋ data race; TOCTOU/order-violation — гонки БЕЗ единой разделяемой memory location
- **Формулировка:** race condition (исход зависит от тайминга/порядка) — более широкое понятие, чем data race (конкурентный доступ к одной memory location, ≥1 запись, без синхронизации). Не всякая гонка — data race: TOCTOU (check-then-act) и order violation — race condition без классической единой memory location. Практическое следствие: их нельзя «починить одним локом вокруг общего объекта» (A1, A4, D1).
- **Scope (A.2.6 ClaimScope):** несёт вес на этапе КЛАССИФИКАЦИИ бага и ВЫБОРА примитива в потоковом Python-коде bounded context (локи вокруг сокета, буферы кадров по потокам-владельцам). НЕ несёт вес для single-threaded кода и для чисто межпроцессной/распределённой согласованности (за non-use boundary — happens-before Лампорта S7 сознательно оставлен как фон). **Фальсификатор:** если бы КАЖДЫЙ race condition в потоковом Python устранялся добавлением одного лока вокруг shared memory — различение было бы избыточно. Order violation (поток B читает раньше, чем A инициализировал) показывает обратное — лок не создаёт отсутствующую happens-before связь.
- **Анти-тезис (NQD ≥3):**
  1. «Под GIL для чистого Python-кода настоящих data race нет (нет разрыва слова, нет reordering видимости на уровне bytecode) — значит различение data race vs race condition для нас академическая роскошь.» — Контраргумент: это довод ЗА различение, а не против. Именно потому, что data-race-подкласс под GIL почти пуст для pure Python, инструменты и интуиция из C/Java (TSan «защити каждую memory location») промахиваются мимо реального класса — TOCTOU/atomicity violation над СОСТАВНОЙ операцией. Различение меняет, какие средства применимы (для pure Python нет TSan-эквивалента — D4).
  2. «TOCTOU — это про файловую систему и безопасность, тащить security-термин в threading = category error.» — Контраргумент: TOCTOU — общий класс check-then-act (A4), инстанцируется и внутри процесса без ФС: `if key not in d: d[key]=...`, ленивая инициализация `if x is None: x = create()`. Привязка только к ФС — необоснованное сужение scope; Netzer/Miller (S5) разделяют формально независимо от носителя.
  3. «Data race и race condition на практике синонимы — разработчик в обоих случаях ставит лок, разница ничего не меняет.» — Контраргумент: НЕ в обоих. Order violation (D1) — race condition БЕЗ конкуренции за один лок; лечится не локом вокруг ресурса, а установлением happens-before (`Event`/`join`/`Condition`). Слияние терминов ведёт к НЕВЕРНОМУ ремонту (лок там, где нужна синхронизация порядка).
- **Тип связи:** scope-dependent (активен на этапе классификации бага/выбора примитива); composes с Тезисом 6 (order violation — частный случай «неверного допущения о порядке»); conflict с «любая гонка = добавь лок».

### Тезис 2 — GIL атомизирует ОДИН bytecode, НЕ составную операцию
- **Формулировка:** GIL гарантирует атомарность одной bytecode-инструкции и отдельной встроенной операции контейнера (`list.append`, `dict.__setitem__`), но НЕ последовательности (`x += 1`, `d[k]=d[k]+1`, `if k not in d: d[k]=v`) — это несколько bytecode, между которыми интерпретатор переключает поток. Read-modify-write и check-then-act остаются гонкой (B1, B3, D1). ME1: `counter += 1` в двух потоках даёт < 200000.
- **Scope (A.2.6):** специфично **CPython с GIL** (эталонная реализация). НЕ переносится на PyPy/Jython/GraalPy/free-threaded build без переверификации (B4/B5). **Фальсификатор:** если бы составная RMW-операция без лока давала корректный результат под нагрузкой двух потоков — тезис ложен; ME1 — эмпирический фальсификатор в подтверждающую сторону.
- **Анти-тезис (NQD ≥3):**
  1. «На CPython с GIL операции `dict`/`list` потокобезопасны — лишние локи только вредят (lock convoy, B2).» — Контраргумент: атомарна ОТДЕЛЬНАЯ операция, не последовательность. `d[k] = d[k] + 1`, `if k in d: del d[k]`, `q.append(x)`+`q.pop()` через check — составные, гонка сохраняется (B3, D1). Ложная уверенность «использую dict → потокобезопасно» — прямой источник atomicity violation, а не защита от convoy.
  2. «Даже если это несколько bytecode — вероятность переключения ровно между ними ничтожна (switchinterval 5 мс против наносекунд операции), на практике не проявится.» — Контраргумент: (а) переключение по таймеру — НЕ единственный safepoint: вокруг I/O GIL освобождается в непредсказуемых точках (B2), а наш bounded context — сетевой I/O (сокет AMQP), где составное состояние часто держат ВОКРУГ блокирующего вызова; (б) «низкая вероятность» — это редко-воспроизводимый баг, а не его отсутствие; редкость УХУДШАЕТ диагностику (D3: случайный stress-test плохо ловит именно такой интерливинг), не устраняет дефект.
  3. «PEP 703 (free-threaded) скоро сделает это неважным — там своя внутренняя синхронизация объектов, локи руками не нужны.» — Контраргумент: НАОБОРОТ (B4) — под free-threaded инварианты B1/B3 перестают быть «бесплатной» гарантией; код, полагавшийся на GIL как неявный global lock, ломается СИЛЬНЕЕ и требует переаудита. Плюс temporal decay: точный статус free-threaded (когда default) — самое молодое и неверифицированное знание пакета (2023); «скоро неважно» = temporal ambiguity, не аргумент.
- **Тип связи:** composition с Тезисом 6 (та же false-atomicity-ассумпция — корень D1); conflict с «GIL = бесплатный мьютекс»; scope-dependent (истинно только для GIL-сборки CPython).

### Тезис 3 — Deadlock ⟺ 4 условия Коффмана; снять любое одно (обычно total lock order) — исключить структурно
- **Формулировка:** deadlock ⟺ одновременно mutual exclusion + hold-and-wait + no preemption + circular wait. Достаточно нарушить ЛЮБОЕ одно; чаще всего вводят total order на захват локов, убирая circular wait (A2).
- **Scope (A.2.6):** применим к любому lock-based коду с ≥2 локами. НЕ к чисто lock-free/ownership-transfer дизайну (нет mutual exclusion → deadlock невозможен by construction, но появляется свой класс — livelock/ABA). **Фальсификатор:** если ввести строгий порядок захвата всех локов и deadlock всё равно возникает — значит в цикле ожидания есть не-lock ресурс (напр. `join` на поток, который ждёт тебя), и модель «только 4 условия на локах» неполна для этого случая.
- **Анти-тезис (NQD ≥3):**
  1. «У нас один лок вокруг сокета — circular wait с одним локом невозможен, deadlock исключён, дисциплина порядка не нужна.» — Контраргумент: (а) reentrancy — тот же поток повторно берёт нерекурсивный `Lock` → self-deadlock БЕЗ circular wait между потоками (решение — `RLock` или не-повторный захват, а не порядок); (б) один лок сегодня → два завтра (лок сокета + лок буфера кадров в нашем же bounded context); дисциплина порядка превентивна. «Один лок» не гарантирует «навсегда один».
  2. «Total lock ordering дорого и хрупко (нужен глобальный порядок всех локов); проще timeout на acquire + retry.» — Контраргумент: timeout-based backoff превращает deadlock в **livelock/starvation** (два потока бесконечно берут-отпускают-ретраят) — это замена одного failure mode другим, не устранение. A2: total order убирает circular wait СТРУКТУРНО (гарантия); timeout убирает no-preemption ценой прогресса (вероятностное смягчение). Silent fusion этих двух гарантий недопустим.
  3. «4 условия Коффмана — про ОС-мьютексы; в Python с GIL и высокоуровневыми примитивами (`queue.Queue`) они неприменимы.» — Контраргумент: условия язык-независимы (A2); `queue.Queue` ВНУТРИ держит `Lock`+`Condition`, deadlock достижим (bounded queue: producer ждёт свободное место под условием, которое разблокирует только consumer, заблокированный на другом локе). Высокоуровневость ПРЯЧЕТ локи, не отменяет условия.
- **Тип связи:** scope-dependent (активен при ≥2 локах или reentrancy); conflict с «один лок → расслабься» и с «timeout = решение deadlock»; composes с Тезисом 5 (ownership-transfer убирает mutual exclusion как класс).

### Тезис 4 — Guarded Suspension: `cond.wait()` ОБЯЗАН быть в `while`, перепроверяющем предикат
- **Формулировка:** ожидание условия на condition variable — в цикле `while not predicate(): cond.wait()`, не в `if`. Две независимые причины: (а) spurious wakeup; (б) notify мог относиться не к тебе (несколько ждущих на разные предикаты, `notify_all` будит всех) / элемент забрали первым между пробуждением и повторным захватом лока (A3, C2).
- **Scope (A.2.6):** применим к ожиданию ПРОИЗВОЛЬНОГО предиката на `threading.Condition`. НЕ нужен там, где примитив сам инкапсулирует ожидание (`queue.Queue.get`, одноразовый `Event.wait`). **Фальсификатор:** если убрать `while` и под нагрузкой с несколькими ждущими + `notify_all` ни разу не продолжить с ложным предикатом — но отсутствие наблюдения ≠ корректность (D3), фальсификатор здесь слабый по своей природе (баг редко-воспроизводим).
- **Анти-тезис (NQD ≥3):**
  1. «Python `threading.Condition` не даёт spurious wakeup (это про pthread/Java) — значит `if` достаточно.» — Контраргумент: даже приняв отсутствие spurious wakeup, ВТОРАЯ причина `while` остаётся независимо (C2): `notify_all` будит всех ждущих с разными предикатами, «настоящий» notify мог быть не тебе, а другой поток мог забрать элемент первым между твоим пробуждением и повторным захватом лока. Аргумент бьёт лишь по одной из двух причин.
  2. «Проще заменить `Condition` на `queue.Queue` — там while-предикат уже внутри, думать не надо.» — Контраргумент: часто верно и хорошо (это Тезис 5), НО `Queue` решает только «ждать элемент FIFO»; не покрывает произвольный/составной предикат («соединение восстановлено И буфер дренирован» — наш `DrainGuard`). Подмена `Condition`→`Queue` СУЖАЕТ выразительность; для не-FIFO-условий прямого аналога нет.
  3. «Достаточно `if predicate: wait()` + одна строка ручной перепроверки после — `while` избыточен.» — Контраргумент: ручная одноразовая перепроверка при повторно-ложном предикате снова пропускает (`if x: wait(); <continue>` едет с ложным x, а `assert x` — падает) — это `while`, развёрнутый НЕПРАВИЛЬНО (не зациклен). По A.11 (нельзя вычесть цикл без потери) `while` — минимальная корректная форма; экономия строки = скрытый баг.
- **Тип связи:** composition с claim A3 (Hoare monitor: condition variable не хранит состояние → обязателен re-check); conflict с «`if` достаточно»; scope-dependent (только для произвольного предиката, не для инкапсулированного ожидания).

### Тезис 5 — Thread confinement / ownership transfer дешевле, чем shared-state + lock везде
- **Формулировка:** самый дешёвый способ избежать гонки — не делить mutable-объект, а передавать его владение ЦЕЛИКОМ через thread-safe очередь, так что в любой момент им владеет ровно один поток. Синхронизационная поверхность сжимается до одной уже-потокобезопасной точки передачи (put/get) вместо N залоченных обращений (C3, ME5). Прямая инстанциация в bounded context: буферы кадров по потокам-владельцам.
- **Scope (A.2.6):** применим когда состояние ДЕЛИМО на владеемые-одним-потоком части. НЕ применим к неразделимому единому ресурсу (один fd сокета) — там сериализация локом (C1) / Half-Sync/Half-Async (C5). **Фальсификатор:** если объект после передачи всё ещё мутируется обоими потоками (producer не «отпустил») — confinement нарушен, гонка вернулась; наблюдаемо (два writer'а к одному буферу).
- **Анти-тезис (NQD ≥3):**
  1. «Передача через очередь — тоже разделяемое состояние (сама очередь); гонку просто спрятали в `Queue`, выигрыша против лока нет.» — Контраргумент: выигрыш именно в СОКРАЩЕНИИ синхронизационной поверхности до одной уже-потокобезопасной границы (put/get локается один раз внутри) вместо N мест, каждое из которых легко забыть залочить (D1 atomicity violation = «забыл в одном месте»). Аудитору проверить одну границу, а не N обращений — меньше поверхности ошибки, не «то же самое».
  2. «В нашем bounded context сокет физически один, все потоки пишут в один fd — confinement неприменим в принципе.» — Контраргумент: confinement применяется НЕ к сокету, а к буферам кадров (каждый буфер — одному потоку-владельцу, прямая C3), а к границе сокета — отдельный лок / Half-Sync-Half-Async (C5: I/O-слой сериализован, обработка confined). Confinement — не «всё или ничего», а разбиение состояния на confined и shared части; «неприменим целиком» = ложная дихотомия.
  3. «Ownership работает в Go/CSP и Rust, потому что там move-семантика by design; в Python объект после `put()` физически доступен producer'у — дисциплина держится на честном слове.» — Контраргумент: верно, что Python не форсит move (реальная граница — см. контрпример 4). НО альтернатива (лок на каждое обращение) ТОЖЕ держится на честном «не забудь взять лок»; оба на дисциплине, но confinement даёт МЕНЬШУЮ поверхность дисциплины (одна конвенция «не трогать после put», проверяемая ревью/линтом/тестом, против N мест захвата). Отсутствие compiler-enforcement — довод за тест/линт, не против паттерна.
- **Тип связи:** composition с C5 (Half-Sync/Half-Async — confined обработка за queueing-границей) и с Тезисом 3 (убирает mutual exclusion как класс → deadlock невозможен для confined части); scope-dependent (делимость состояния); conflict с «share memory + lock везде».

### Тезис 6 (центральный, ИИ-специфичный) — Реальные баги — это неверные ДОПУЩЕНИЯ об атомарности/порядке, а не «забыл лок»; ревьюить инвариант, не наличие лока; LLM воспроизводит те же допущения чаще, а TSan для pure Python нет
- **Формулировка:** эмпирически доминируют atomicity violation и order violation (D1); корень — не «забыл поставить лок», а неверное ментальное допущение об инварианте (D2). Ревью эффективнее, когда проговаривает «какой инвариант это защищает». LLM генерирует синтаксически-корректный конкурентный код, но статистически чаще ошибается в тонком (if-vs-while, exception-safety лока, compound-atomicity, ленивая init без лока — AI2), потому что обобщает по поверхностному сходству кода, а не по инварианту; для pure Python нет TSan-эквивалента (D4, AI3), поэтому ИИ гэп тулинга НЕ закрывает — только ускоряет производство кода, который надо проверять тем же чек-листом (D3+D2).
- **Scope (A.2.6):** применим к РЕВЬЮ/ДИАГНОСТИКЕ потокового кода (человеческого и LLM-сгенерированного). НЕ отвечает на «есть ли вот здесь конкретная гонка» (это требует прогона/анализа, не общего принципа). ИИ-часть **максимально decay-чувствительна**: агентные инструменты с race-детектором в цикле могут это изменить; AI2 помечен `hypothesis`, не факт. **Фальсификатор:** если для pure Python появится зрелый TSan-эквивалент в стандартном тулчейне — AI3/D4-часть устареет; если контролируемое измерение покажет, что LLM НЕ чаще людей ошибается в этих местах — AI2 опровергнут (но D1/D2-основание чек-листа устоит).
- **Анти-тезис (NQD ≥3):**
  1. «Раз корень — неверное допущение, а не отсутствие лока, то лок ВЕЗДЕ ("на всякий случай") безопасно решает проблему.» — Контраргумент: over-locking даёт свои failure mode — deadlock (Тезис 3), lock convoy (B2) — и МАСКИРУЕТ order violation (лок вокруг ресурса не создаёт отсутствующую happens-before связь, D1). Плюс «лок везде» не документирует ИНВАРИАНТ, который защищает, а D2 требует именно этого. Лишний лок = ложное чувство защиты + новый класс багов.
  2. «У LLM-ассистента есть доступ к запуску кода/тестов — он закроет гэп (нет TSan) прогоном тестов, в отличие от человека прошлого.» — Контраргумент: прогон тестов = D3 (stress-test), который плохо ловит редкий интерливинг даже с уменьшенным switchinterval; LLM генерирует код текстом, не symbolic execution/model checking — он не «видит» гонку статически, как TSan для C. Наличие тула-раннера УСКОРЯЕТ тот же слабый метод, не заменяет отсутствующий детектор; агентный самопрогон ещё и рискует self-agreement (тест написан той же сессией под то же ошибочное допущение).
  3. «AI2 ("LLM чаще ошибается в тонком") — это `hypothesis`, а не измеренный факт (так помечено в sota-research); строить на нём чек-лист ревью = опираться на мнение.» — Контраргумент (адверсарный ход против собственного source-pack): чек-лист (AI4) опирается НЕ на AI2 в одиночку, а на ПЕРЕСЕЧЕНИЕ AI2 (hypothesis) и D1/D2 (эмпирический факт: atomicity/order violation доминируют, ASPLOS 2008). Снять AI2 полностью — и D1/D2 всё равно обосновывают те же проверки для ЛЮБОГО автора. Чек-лист durable по человеческому основанию; AI-специфика лишь повышает ПРИОРИТЕТ, не является единственной опорой. Именно так `hypothesis` не повышается до `fact` молча.
- **Тип связи:** composition с Тезисом 2 (та же false-atomicity-ассумпция) и Тезисом 1 (order violation = race без общего лока); requires refresh-триггеры G.11 (AI-claims и статус тулинга decay-чувствительны); conflict с «лок везде = безопасно» и «LLM закроет гэп сам».

## 3. Контрпримеры (похоже, но НЕ применять — A.11 Sharp Boundary)

> Отдельно от анти-паттернов (те — «как НЕ исполнять»): контрпример = случай, внешне подпадающий под тезис, но лежащий ЗА границей его scope. Одно-предложный inclusion/exclusion-тест (A.11:2, гейт Sharp boundary).

1. **Double-checked locking: Java-падение vs CPython (граница Тезиса 2 / claim C4).** Классическая сломанная DCL из Java/C++ ВЫГЛЯДИТ как та же угроза в Python, но падает по ДРУГОЙ причине — reordering видимости под настоящей memory model (JMM до JSR-133), которой у GIL-CPython по сути нет (bytecode сериализован, B5). Граница: урок «неверифицированная оптимизация синхронизации» — переносится; конкретный МЕХАНИЗМ падения (visibility reordering) — нет. Тест: «я переношу УРОК про неверифицированность или конкретный механизм визибилити? Если механизм — это silent fusion двух разных memory model».

2. **GIL-атомарность `list.append` vs составная последовательность (граница Тезиса 2).** `q.append(x)` в одном потоке и `q.pop()` в другом — каждая атомарна (B3), ВЫГЛЯДИТ как «уже потокобезопасная очередь», но `if q: x = q.pop()` — составная (check-then-act, между проверкой и pop другой поток опустошил). Граница: атомарность применима к ОДНОЙ встроенной операции; к паре «проверить-и-взять» — нет. Тест: «одна встроенная операция или последовательность из двух?»

3. **`setdefault` / атомарный write-if-absent vs дорогой побочный эффект (граница Тезиса 1 / A4).** `cache.setdefault(k, expensive())` ВЫГЛЯДИТ как атомарное устранение TOCTOU, но устраняет только гонку за ЗАПИСЬ; `expensive()` вычисляется каждый раз ДО вызова (ME3) — гонку за повторное вычисление не убирает. Граница: атомарный write-if-absent ≠ атомарный compute-once. Тест: «что должно быть атомарным — запись или вычисление? Если вычисление — нужен лок вокруг всей секции, не `setdefault`».

4. **Rust/Go ownership vs Python confinement (граница Тезиса 5).** «Передал владение — трогать нельзя» ВЫГЛЯДИТ как Rust move / Go «share memory by communicating», но Python НЕ форсит: объект после `put()` физически доступен producer'у, компилятор не запретит мутацию. Граница: в Rust ownership enforced типами (borrow-checker); в Python — конвенция, держится ревью/линтом/тестом, не языком. Тест: «форсит ли язык move, или это дисциплина? Если дисциплина — нельзя ссылаться на невозможность-как-в-Rust, нужен тест/линт».

5. **Уменьшение `sys.setswitchinterval` в тесте vs настоящий race detector (граница Тезиса 6 / D3/D4).** Уменьшить switchinterval в стресс-тесте ВЫГЛЯДИТ как «поймаем гонку как TSan», но это лишь повышает ЧАСТОТУ переключений — не перечисляет интерливинги систематически (CHESS) и не сравнивает happens-before/vector clocks (TSan). Граница: вероятностное учащение ≠ систематическое/статическое доказательство отсутствия гонки. Тест: «я увеличиваю вероятность попасть в окно гонки или ДОКАЗЫВАЮ её отсутствие? Первое — не гарантия».

6. **Race condition vs безобидный недетерминизм порядка (граница Тезиса 1).** Два потока пишут в лог/метрику в непредсказуемом порядке — ВЫГЛЯДИТ как race condition, но если НИКАКОЙ инвариант не зависит от этого порядка, это benign недетерминизм, не баг (и лок здесь — лишняя сериализация → convoy). Граница: race condition — когда КОРРЕКТНОСТЬ зависит от порядка; безобидный недетерминизм — когда порядок лишь наблюдаемо разный. Тест: «зависит ли инвариант от порядка, или порядок просто наблюдаемо разный?»
