## Bounded context (A.1.1)

Конкурентное программирование **в многопоточном (не asyncio) Python-коде**, где несколько ОС-потоков
делят один ресурс с явным жизненным циклом (сокет/соединение, канал, буфер данных) и должны
координироваться без порчи состояния и без взаимной блокировки.

Референсная инстанциация — `kombu-pyamqp-threadsafe` (проверено чтением исходников,
`src/kombu_pyamqp_threadsafe/__init__.py`, 2026-07-14):
- **Локи вокруг сокета/соединения**: `threading.RLock` (`_transport_lock`,
  `_connection_dispatch_lock`, `_create_channel_lock`) — защита операций над общим
  AMQP-соединением/сокетом от нескольких потоков-продюсеров/потребителей.
- **Выбор дренера / координация "кто сейчас читает сокет"**: класс `DrainGuard` —
  `threading.Condition` + вспомогательный `RLock`, паттерн "single active reader,
  остальные ждут результат", с `start_drain()/finish_drain()/wait_drain_finished()`,
  явными assert-инвариантами («нельзя ждать свой же дрейн — deadlock», «нельзя завершить
  чужой дрейн»), late-dispatch буферизацией и защитой от гонки «drain уже начался, но
  `_drain_is_active_by` ещё не выставлен» (комментарий в коде: порядок операций внутри
  лока критичен для корректности).
- **Буферы кадров/каналы по потокам-владельцам**: `channel_thread_bindings` — привязка
  channel_id к `threading.get_ident()` владельца, явный `change_owner()` при смене потока,
  `threading.local()`-подобная модель "канал не потокобезопасен сам по себе, но владение
  можно передать управляемо".
- **Событие для teardown**: `threading.Event` (`_teardown_event`) — сигнал "пора остановиться"
  без ручного опроса.

## Intended reader

Роль **architect** (и dev при реализации/ревью), которая:
- проектирует или ревьюит многопоточный Python-код с разделяемым состоянием
  (не обязательно AMQP/kombu — любой похожий домен: пул соединений, кэш, буфер очереди);
- должна классифицировать race condition, выбрать примитив синхронизации, обосновать
  порядок захвата локов, распознать TOCTOU/check-then-act до того, как это стало инцидентом.

## First use

Первая полезная задача, которую DPF должен решать: **дать architect/dev словарь и чек-лист
для ревью/проектирования конкурентного участка кода** — "к какому классу гонки относится
этот код, какой примитив уместен, что проверить перед merge" — до того, как баг проявится
в проде (флаки в тестах, redundant drain, двойное чтение сокета, потеря события).
Конкретно для kombu-pyamqp-threadsafe: обосновать/провалидировать существующие решения
(RLock вокруg transport, Condition-based DrainGuard, per-thread channel ownership) через
призму устоявшихся паттернов индустрии, и дать язык для будущих подобных решений.

## Non-use boundary

- **НЕ asyncio/корутины** — event loop, `async/await`, single-threaded cooperative
  concurrency — отдельная модель (нет data race в классическом смысле между await-поинтами
  внутри одного loop; другие failure modes). Если код мигрирует на asyncio — нужен другой DPF.
- **НЕ межпроцессная конкуренция** — `multiprocessing`, разделяемая память между процессами,
  file locks между процессами — другой набор примитивов и гарантий (нет GIL, нет общей кучи).
- **НЕ распределённая согласованность** — консенсус, репликация, CAP-теорема, распределённые
  локи (Redis/Zookeeper) — другой домен (сетевые партиции, а не потоки одного процесса).
- **НЕ AMQP-протокол как таковой** (семантика фреймов, ack/nack, exchange routing) — это
  предметная область kombu/AMQP, а не конкурентное программирование; DPF описывает КАК
  синхронизировать доступ к общему ресурсу, а не ЧТО это за ресурс.
