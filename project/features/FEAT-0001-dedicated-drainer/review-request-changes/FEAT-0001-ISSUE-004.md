# ISSUE-004 — Busy-spin `_read` при `timeout=0` на фрагментированном кадре: wakeup не срабатывает, `_transport_lock` держится

- **Серьёзность:** major
- **Файлы:**
  - `src/kombu_pyamqp_threadsafe/__init__.py:743-763` (`_drain_into_buffers` — `super().drain_events(timeout=0)` под `_transport_lock`)
  - `.venv/lib/python3.14/site-packages/amqp/transport.py:625-647` (`TCPTransport._read` — busy-spin на EAGAIN для не-initial чтения)
  - `src/kombu_pyamqp_threadsafe/drainer.py:294-306` (self-pipe wakeup эффективен только внутри `sel.select`, не внутри чтения)

## Класс ошибки

DPF-DEDICATED-IO-THREAD Pattern 6, чек-лист п.1: «wakeup будит поток ВНУТРИ poll, не ждёт таймаута». Здесь wakeup покрывает только `select()`, но НЕ блокирующее/крутящееся чтение сокета — поток может застрять вне точки, где self-pipe его достаёт. Плюс удержание `_transport_lock` поперёк потенциально долгого чтения (то, ради устранения чего затевалась вся фича — см. PLAN §«Главная боль»).

## Описание

`_drain_into_buffers` читает `super().drain_events(timeout=0)` под `_transport_lock`. При `timeout=0` сокет переводится в неблокирующий режим, и `TCPTransport._read` (amqp) для продолжения кадра (`initial=False`) на `EAGAIN` делает `continue` в `while len(rbuf) < n` **без выброса таймаута** (таймаут поднимается только на initial-чтении 7-байтного заголовка, `transport.py:634-637`). Если брокер прислал заголовок кадра, но тело пришло частично (фрагментированный/throttled кадр — реалистично под нагрузкой, воспроизводится toxiproxy bandwidth), дренер **крутит CPU в busy-spin**, пока не приедет остаток тела.

Две конкурентные беды на время этого спина:
1. **Wakeup не работает.** Поток не в `sel.select`, а в `recv`-цикле. `stop()`/`close()`, записавшие байт в self-pipe, его не прерывают — джойн истечёт по таймауту (`drainer.py:251-254` залогирует warning), поток продолжит спин.
2. **`_transport_lock` держится.** Паблишеры (`frame_writer`, `__init__.py:645`) и `close()` (шлёт Connection.Close через тот же `frame_writer`) блокируются на локе до конца чтения. Худший случай — пир прислал частичный кадр и «замолчал»: спин + удержание лока тянутся до TCP-ошибки половинно-открытого соединения (могут быть минуты).

Это ровно тот класс «лок держится поперёк блокирующего чтения», который фича должна была устранить, — здесь он воспроизводится точечно на фрагментированном кадре и вдобавок нейтрализует заявленный wakeup-примитив.

## Доказательство (evidence)

Чтение исходника `amqp/transport.py:625-647`: цикл `while len(rbuf) < n:`; `recv` на неблокирующем сокете кидает `OSError(EAGAIN)`; ветка `if exc.errno in _errnos:` → `if initial and self.raise_on_initial_eintr: raise socket.timeout()` иначе `continue`. Для `initial=False` (продолжение кадра) выхода по таймауту нет → busy-spin до прихода данных. План уже помечал этот риск (§8, TDD-итерация №10 «фрагментированные кадры»), но как латентность/CPU; конкурентное следствие (срыв wakeup + блокировка `_transport_lock`) в чек-листе wakeup не отражено.

## Направление фикса (без кода)

Обсудить с заказчиком (это про идею, не только реализацию): busy-spin — свойство `_read` при неблокирующем сокете, его нельзя убрать точечной правкой дренера. Варианты направления:
- покрыть сценарий нагрузочным тестом (toxiproxy bandwidth, сообщение ~`frame_max`) до перевода дефолта, как и планировалось (№10), и замерить фактические удержание лока/CPU;
- рассмотреть чтение с малым НЕнулевым таймаутом на продолжении кадра вместо чистого `timeout=0`, либо гейт «сокет читаем целиком» перед взятием лока;
- зафиксировать в чек-листе wakeup честный вердикт: примитив будит `select()`, но не прерывает чтение — это ограничение п.1, а не полный pass.
