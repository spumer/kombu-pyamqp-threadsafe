# Hub из kombu.asynchronous как замена lock-based подхода в kombu-pyamqp-threadsafe

Исследование по kombu 5.6.2 и py-amqp 5.3.1 (site-packages в `.venv/lib/python3.14/site-packages/`), наша реализация — `src/kombu_pyamqp_threadsafe/__init__.py`. Все спорные утверждения прошли адверсариальную проверку по исходникам; опровергнутых не оказалось.

---

## 1. TL;DR

**Нет, Hub «как есть» нашу thread-safe реализацию не заменяет.** Hub — это только read-side: poller + таймеры + очередь `call_soon`. Публикация, RPC-вызовы (`queue_declare`, confirms и т.д.) и переподключение через Hub не проходят вообще — это синхронные блокирующие операции на потоке вызывающего (`kombu/transport/pyamqp.py:217-219` регистрирует сокет только на чтение; publish = `sock.sendall` напрямую, `amqp/transport.py:621`).

**Архитектура «один IO-поток владеет сокетом, остальные потоки отдают ему работу» — здравая** (именно её рекомендует автор Celery в py-amqp#267 вместо мьютексов), но чтобы построить её на Hub, придётся написать с нуля: wakeup-механизм, submit-API с future'ами, промисный (недренящий) RPC-слой вместо `wait()`, корреляцию publisher confirms по delivery tag (в py-amqp её нет), реконнект. Это переписывание thread-safety-слоя сопоставимой или большей сложности поверх компонента в maintenance mode.

**Что делать:** полную миграцию не делать. Есть рентабельный гибридный шаг — выделенный поток-дренер на базе Hub (или без него), который убирает главную проблему текущего дизайна: паблишеры перестают ждать `_transport_lock`, пока дренер сидит в блокирующем чтении. Детали в §8.

## 2. Как устроен Hub

Hub (`kombu/asynchronous/hub.py`, 403 строки) — event loop с **единственным потоком-владельцем**, построенный на генераторе:

- `create_loop()` возвращает генератор (`hub.py:292-385`); одна итерация = выгрузить `_ready` (колбэки `call_soon`), выполнить наступившие таймеры через `fire_timers` (это же вычисляет таймаут poll, `hub.py:139-160`), прогнать `on_tick`, вызвать `poll(poll_timeout)` (`hub.py:321`) и раздать read/write-колбэки по fd.
- `run_once()` = `next(self.loop)` (`hub.py:198-202`); `run_forever()` крутит `run_once` до `Stop` (`hub.py:187-196`). **Сам kombu цикл никогда не запускает** — им владеет встраивающее приложение (в Celery это `asynloop` в `celery/worker/loops.py`, крутящий `next(loop)` в MainThread воркера).
- Poller выбирается в `kombu/utils/eventio.py:312-324`: epoll на Linux, `select.poll` на macOS (kqueue включается только на NetBSD).

**Потокобезопасность: единственная безопасная извне операция — `call_soon`.** Единственный лок во всём модуле — `_ready_lock` (`hub.py:86`), защищающий только множество `_ready` в `call_soon` (`hub.py:204-209`) и `_pop_ready` (`hub.py:252-256`). Всё остальное без синхронизации: `readers`/`writers` — обычные dict'ы (`hub.py:81-82`), `add()` мутирует poller и dict'ы без лока (`hub.py:168-181`), `remove_reader`/`remove_writer` — неатомарные многошаговые последовательности (`hub.py:226-244`), куча таймера — голый heapq без лока (`timer.py:171-173`, `timer.py:190-219`), `LaxBoundedSemaphore` — незащищённый read-modify-write (`semaphore.py:45-90`). Вдобавок генератор кэширует `poller.poll`, `readers`, `writers` как локальные переменные при создании (`hub.py:296-303`) — `reset()` из другого потока оставляет цикл с протухшими ссылками.

**Wakeup-механизма нет.** Ни self-pipe, ни eventfd, ни socketpair (проверено grep'ом по всему hub.py и пакету). `call_soon` из чужого потока лежит невидимым, пока `poll()` не вернётся сам — по умолчанию таймаут 1 секунда, когда таймеров нет (`hub.py:312`), плюс idle-ветка `sleep(min(timeout, 0.1))` при отсутствии fd (`hub.py:382-384`). Задержка до 1 с на любую cross-thread-операцию.

**Владение сокетом:** один колбэк на fd на направление; готовый fd без колбэка молча выкидывается из цикла (`hub.py:337-345, 359-361`). Поток цикла — единственный легитимный читатель зарегистрированного сокета.

## 3. Что готово (работает уже сейчас)

1. **Read-side pump для pyamqp — production-proven в Celery.** `Transport.register_with_event_loop` делает `loop.add_reader(connection.sock, self.on_readable, connection, loop)` и ставит `raise_on_initial_eintr = True` (`kombu/transport/pyamqp.py:217-219`). Колбэк `on_readable` (`kombu/transport/base.py:244-248`) через `_make_reader` (`base.py:222-239`) вызывает `drain_events(timeout=0)`, при успехе перепланирует сам себя через `loop.call_soon(_read, loop)`, глотает `socket.timeout`/EAGAIN/EINTR.

2. **`read_frame` резюмируемый** и годится как readable-колбэк: при `socket.timeout` частично прочитанные байты возвращаются в `self._read_buffer` (`amqp/transport.py:318-336` — pushback на трёх ветках), следующий вызов продолжает с того же места (`amqp/transport.py:282-346`).

3. **Диспетчеризация входящих кадров не привязана к потоку.** `on_inbound_method` маршрутизирует строго по channel_id: `self.channels[channel_id].dispatch_method(...)` (`amqp/connection.py:534-540`); промисы в `_pending` срабатывают независимо от того, какой поток дренил (`amqp/abstract_channel.py:112-159`). IO-поток может пампить кадры, которые завершают ожидания других потоков — сломан только блокирующий wait-конец, не механика промисов.

4. **Таймеры для heartbeat.** `hub.call_repeatedly(interval, conn.heartbeat_check)` (`hub.py:217-218`) — ровно тот паттерн, что использует Celery. `heartbeat_tick` — учёт счётчиков плюс одна запись кадра (`amqp/connection.py:712-776`), безопасен из таймера, если все записи на loop-потоке.

5. **`call_soon` реально потокобезопасен** (`hub.py:86, 204-209`) — валидный, хоть и медленный (до 1 с), примитив маршалинга.

6. **Наш собственный код уже реализует нужную маршрутизацию.** `on_inbound_method` буферизует кадры по каналам вместо inline-диспетчеризации (`__init__.py:591-596`), `_dispatch_pending_events` доставляет только потоку-владельцу канала (`__init__.py:658-666`) — это ровно тот routing-слой, в который гибридный Hub-дренер может складывать кадры без изменений.

## 4. Что можем использовать как есть

- **Hub целиком как внутренний планировщик IO-потока**: `create_loop`/`run_forever`, `fire_timers`, poller через eventio (`hub.py:187-202, 292-385`) — при условии, что ровно один поток его крутит и все `add`/`remove`/таймеры вызываются из него.
- **`pyamqp.Transport.register_with_event_loop` + `base.Transport.on_readable`/`_make_reader`** — весь входящий насос без изменений (`pyamqp.py:217-219`, `base.py:222-248`).
- **`frame_handler` (сборка входящих кадров) и `dispatch_method`** — без изменений, если их вызывает только IO-поток (`amqp/method_framing.py:28-83`, `abstract_channel.py:112-159`).
- **Fast path `frame_writer`** — один `sendall` одного упакованного буфера (`method_framing.py:161-186`); общий bytearray перестаёт быть опасным, когда пишет один поток.
- **`concurrent.futures.Future`** из stdlib как несущий примитив для ответов.
- **Наши `ChannelCoordinator`/`ChannelOperation` и форкнутый `ensure()`** (`__init__.py:442-537, 1023-1097`) остаются валидными в гибридном варианте, где publish остаётся на потоках вызывающих.

## 5. Что требует доработки (для полной single-owner модели)

| # | Что строить | Суть | Сложность |
|---|---|---|---|
| 1 | **Wakeup-механизм** | Hub не видит cross-thread `call_soon` до возврата `poll()` (до 1 с, `hub.py:312, 321, 382-384`). Нужен socketpair/os.pipe, зарегистрированный как reader, + `wake()` вокруг `call_soon`. | Малая (~50 строк), но это семантический форк maintenance-mode компонента — код наш навсегда. |
| 2 | **Submit-API с future'ами** | `submit(op) -> Future` для каждого AMQP-глагола (publish, ack, declare, consume, qos); проброс исключений, таймауты, shutdown. Ничего готового нет. | Средняя. Каждый publish получает 2 переключения контекста вместо одного uncontended RLock сегодня. |
| 3 | **Funnel всех записей на loop-поток** | `frame_writer` — замыкание с одним общим bytearray на соединение (`method_framing.py:109`), bigbody-путь — несколько `write()` на сообщение (`method_framing.py:130-151`). Либо всё пишет loop-поток, либо остаётся write-lock — что обесценивает миграцию. | Средняя (следствие п.2). |
| 4 | **Недренящий RPC** | `AbstractChannel.wait()` регистрирует промис и крутит `self.connection.drain_events(timeout)` до `p.ready` (`abstract_channel.py:86-110`, цикл на :98-99). На loop-потоке это deadlock, вне его — гонка на recv. Нужно `send_method(wait=None)` + ручная привязка `_pending` + завершение Future в колбэке промиса + таймаут через hub-таймер — по каждому семейству методов, с сохранением LIFO-семантики save/restore (`abstract_channel.py:93-95, 105-110`). | **Большая** — фактически приватный async-слой AMQP-клиента. |
| 5 | **Корреляция confirms по delivery tag** | `basic_publish_confirm` завершается на **первом** Ack/Nack канала, delivery tag не сверяется (`channel.py:1806-1823`, pop в `abstract_channel.py:144`). Для конкурентных in-flight confirms нужны future'ы по тегам с обработкой флага `multiple` поверх `_on_basic_ack`/`_on_basic_nack` (`channel.py:150-151`). В py-amqp этой функциональности просто нет. | Большая. |
| 6 | **Доставка в поток-владелец** | Колбэки консьюмера исполняются на Hub-потоке (`kombu/messaging.py:636-668`). Наш контракт (README.md:60-71, Rule 2: ошибки канала поднимаются в потоке-владельце) требует per-channel очередей + wait-циклов на стороне вызывающего — пересборка `channel_frame_buff`/`channel_thread_bindings` по другую сторону границы потоков. | Средняя, но это переписывание того, что уже работает. |
| 7 | **Реконнект** | `base.Transport` кэширует reader-замыкание в name-mangled `__reader` и никогда не инвалидирует (`base.py:182, 244-248`) — после реконнекта оно оборачивает `drain_events` **старого** соединения (Transport-объект переживает реконнект: `kombu/connection.py:972-975` кэширует `_transport`, `_connection_factory` его не трогает). Нужны: снятие старого fd, сброс кэша, ре-регистрация, fail всех pending-future'ов recoverable-ошибками, handshake (сам является drain-циклом, `amqp/connection.py:329-330`) на IO-потоке. Весь класс гонок из фикса 2e2cf64 возвращается как «инвалидация future'ов vs реконнект». | Большая. |
| 8 | **Heartbeats на IO-потоке** | В hub-режиме kombu их не планирует нигде (только ConsumerMixin в drain-режиме, `kombu/mixins.py:188-199`); нужен `hub.call_repeatedly`. | Тривиальная, и строго лучше сегодняшнего (не зависит от каденса дрейна). |
| 9 | **Неблокирующая исходящая запись** | `transport.write` = `sock.sendall` (`amqp/transport.py:621`), для SSL — питоновский цикл (`transport.py:595-609`). Publish под TCP-backpressure на loop-потоке морозит чтение, heartbeats и всех консьюмеров. Правильное решение — исходящий буфер + flush через `add_writer` (метод есть, `hub.py:223-224`, но ни один AMQP-transport его не использует) — это форк выходного пути py-amqp. | Большая. |

## 6. Блокеры / фундаментальные несостыковки

1. **Синхронный RPC-контракт py-amqp.** Каждый `wait()` блокируется в цикле `drain_events` на потоке вызывающего (`abstract_channel.py:86-110`). С loop'ом в другом потоке — гонка двух читателей на recv/`_read_buffer` (разрыв кадров); перенос wait на loop-поток стопорит цикл на broker RTT на каждый RPC и сериализует все app-потоки — конкуренция та же или хуже, чем у наших локов. Async/callback-варианта wait в py-amqp нет.

2. **Hub покрывает только READ для AMQP.** pyamqp не регистрирует writer; publish не касается Hub (`pyamqp.py:217-219`; publish = `send_method` → `frame_writer` → `sock.sendall`, `transport.py:621`). «Hub заменит локи» — категориальная ошибка: Hub никогда не был механизмом сериализации записи.

3. **Confirm-режим структурно несовместим с внешним дренером.** `basic_publish_confirm` публикует и дренит соединение до любого Ack/Nack (`channel.py:1806-1823`); при параллельно дренящем loop-потоке — либо гонка двух читателей, либо неверно атрибутированные confirms (тегов нет). Вдобавок сам publish может читать сокет: `drain_events(timeout=0)` под capability `connection.blocked` (`channel.py:1786-1793`).

4. **Hub single-owner по конструкции и в maintenance mode.** Кроме `call_soon` всё несинхронизировано, wakeup-fd нет, генератор кэширует ссылки при создании (`hub.py:296-303`). История коммитов hub.py с 2018 — только багфиксы (#923, #972, #1489 — лок `_ready` добавлен лишь в 2022, #1587, #1830, #2404); kombu#879 (миграция на asyncio) висит с 2018 без ветки и PR. Мы стали бы единственным production-пользователем неподдерживаемого режима.

5. **Никто в экосистеме не использует Hub для многопоточной публикации.** Celery использует Hub read-only как consumer loop; свои события публикует на **отдельном** write-соединении и флашит их изнутри hub-цикла через `hub._ready.add(self._flush_events)`. Автор Celery (py-amqp#267): «one thread reading/writing from the socket, and the rest of threads communicate with that thread via queue.Queue... if mutexes are used... performance suffers». Прайор-арт (rabbitpy — демонический IO-поток + Queue + socketpair; pika — «not thread-safe», паттерн ioloop-owner + `add_callback_threadsafe`) подтверждает: консенсус — выделенный IO-поток с очередями, а не Hub и не локи.

6. **Разрыв семантического контракта.** README Rule 2 (исключения брокера поднимаются в потоке-владельце канала) не имеет Hub-эквивалента — ошибки всплывают на IO-потоке. Сохранение контракта требует пересобрать per-thread routing, который у нас уже есть, что убивает аргумент «упрощения».

## 7. Сравнение с текущей реализацией

**Что у нас сейчас** (8 примитивов синхронизации): `ThreadSafeConnection._transport_lock` сериализует весь socket I/O — каждый write через обёртку `frame_writer` (`__init__.py:607-631`) и весь реальный дрейн `super().drain_events` (`__init__.py:684-685`); DrainGuard выбирает единственного дренера (`__init__.py:353-436`); кадры буферизуются по каналам и диспетчеризуются pull-ом потоком-владельцем (`__init__.py:591-596, 658-666`).

**Главная боль текущего дизайна:** дренер держит `_transport_lock` на всё время блокирующего чтения — паблишеры стоят до прихода кадра (при `timeout=None` — неограниченно). Плюс один захват глобального RLock на каждый публикуемый кадр даже без конкуренции. Для диагностики этого уже есть LoggingLock с порогом «STUCK» 10 мс (`__init__.py:32, 748-809`).

**Что полная Hub-модель реально выигрывает:**
- (a) порча общего write-буфера и interleaving кадров невозможны **по построению**, а не благодаря локу;
- (b) паблишер никогда не ждёт дренера;
- (c) heartbeats всегда вовремя, независимо от того, дренит ли кто-то;
- (d) кадры каналов, чей владелец не дренит, всё равно пампятся (сегодня лежат в буфере бесконечно).

**Что она лишь переносит, а не убирает:**
- (e) проброс ошибок вызывающему: сегодня бесплатен (вызывающий поток сам исполняет код py-amqp), станет явной future-обвязкой;
- (f) реконнект: не удаляется, а переписывается (класс гонок 2e2cf64 воспроизводится как «pending futures vs teardown»);
- (g) backpressure: блокирующий `sendall` на loop-потоке стопорит **всех** консьюмеров и heartbeats — сегодня стопорятся только другие пользователи сокета; честное решение требует неблокирующего выходного буфера, которого в py-amqp нет.

Итог: сложность перемещается из локов в future-корреляцию, hand-off-очереди, жизненный цикл IO-потока и инвалидацию pending-состояния — объём и тонкость сопоставимы с сегодняшними 8 примитивами.

## 8. Рекомендация

**Полную single-owner миграцию на Hub — не делать.** Это не миграция, а новая клиентская архитектура: приватный async-AMQP-слой поверх замороженного event loop'а, без апстрим-поддержки и без прецедентов использования.

**Идти инкрементально, забрав главный выигрыш дёшево:**

**Шаг 1 (рекомендуется): выделенный поток-дренер вместо DrainGuard-выборов.** Отдельный поток крутит Hub (или просто poll) и владеет только чтением: `poll()` без какого-либо лока; при готовности сокета берёт `_transport_lock` только на `drain_events(timeout=0)` — микросекунды на готовый кадр вместо секунд блокирующего чтения — кадры ложатся в существующий `channel_frame_buff` (`__init__.py:591-596`), после чего condition будит потоки-владельцы для диспетчеризации (`__init__.py:658-666`). Это убирает конкуренцию «дренер держит лок поперёк блокирующего чтения» (`__init__.py:684-685` vs `:613`), не трогая семантику publish/RPC/reconnect/Rule 2 и весь фикс 2e2cf64. Нюанс для контроля: под `timeout=0` частичное тело кадра busy-spin'ится в `_read` (non-initial EAGAIN → `continue`, `amqp/transport.py:630-637`; таймаут поднимается только на initial-чтении 7-байтного заголовка, `transport.py:635-636`) — проверить нагрузочным тестом с фрагментированными кадрами.

**Шаг 2: heartbeat_check на таймер IO-потока** (`hub.call_repeatedly` или свой таймер) — убирает зависимость от каденса дрейна app-потоков.

**Шаг 3 (опциональный, только по результатам бенчмарков после 1-2):** маршрутизация записей через IO-поток с future'ами и снятие write-lock'а. Скорее всего нерентабельно: гибрид уже убирает долгие удержания, а остаётся один uncontended-захват RLock на publish — дешевле двух переключений контекста.

Ключевые файлы: `src/kombu_pyamqp_threadsafe/__init__.py` (наш слой), `.venv/lib/python3.14/site-packages/kombu/asynchronous/hub.py`, `kombu/transport/pyamqp.py`, `kombu/transport/base.py`, `amqp/transport.py`, `amqp/method_framing.py`, `amqp/abstract_channel.py`, `amqp/channel.py`, `amqp/connection.py`.
